# 驴迹总监汇报页 OST 替换 Design

日期：2026-03-08

## 目标

优化 [lvji-director-report.html](/Users/wenjiayan/PycharmProjects/驴迹AI/lvji-director-report.html) 的 `P0 / 关键场景` 区块：

1. 去掉“关键旅程 6 节点”和“验证闭环”两组内容。
2. 增加 `Opportunity Solution Tree`。
3. 保持 `Boardroom Atlas` 风格一致，并让新模块更适合总监汇报扫读。

## 设计判断

当前 `关键场景` 已经有三张场景卡和角色价值条。继续保留“旅程 + 验证”会让页面后半段偏方法论，不够像高层汇报页。

对总监而言，更有价值的是：

1. 目标是什么。
2. 机会点是什么。
3. 每个机会对应什么方案。
4. 先做什么实验验证。

因此，这一块应从“过程说明”改成“战略机会树”。

## 视觉方向

采用 `Atlas Tree` 风格。

### 概念

把 `Opportunity Solution Tree` 做成地图式战略树，而不是普通 Mermaid 或表格。

### 视觉结构

1. 四列横向展开：`目标 -> 机会 -> 方案 -> 实验`。
2. 每一列使用独立栏目卡，保留纸面简报感。
3. 各卡片之间使用细连接线与编号提示，形成“战略枝干”感。
4. `目标` 列使用单张主节点卡，视觉最重。
5. `机会 / 方案 / 实验` 逐列展开，视觉权重逐步减弱。

### 风格约束

1. 继续使用现有 `paper / ink / accent / signal` 变量。
2. 连接线采用路线图式虚线或细实线，不做复杂动画。
3. 卡片保持当前圆角、纸面纹理和高层简报气质。
4. 不引入深色科技风，不做 Mermaid 默认样式。

## 内容结构

### 目标

`目标：让 AI 成为驴迹新的增长与效率引擎`

### 机会

1. 内部交付和售前知识分散
2. 游客服务依赖人工与静态内容
3. 景区营销与运营动作反应慢
4. 客户数据和项目知识难沉淀

### 方案

1. 数字交付员工
2. 数字导游员工
3. 数字营销员工
4. 数字运营员工
5. 文旅知识中枢与主数据治理

### 实验

1. 10 份历史方案自动生成与人工对比
2. 1 个景区游客问答 Shadow mode
3. 1 个活动周期内容生产 A/B 测试
4. 节假日日报与异常预警试点
5. 1000 条知识入库与准确率评估

## 布局细节

### 桌面端

1. 四列并排展示。
2. 左到右阅读，符合总监汇报节奏。
3. 方案列和实验列可容纳更多卡片。

### 平板端

1. 两列布局。
2. 目标与机会在上，方案与实验在下。

### 手机端

1. 单列堆叠。
2. 保留列标题，连接关系用文案顺序而不是视觉连线表达。

## 预期结果

优化后，这一部分会从“解释过程”变成“组织战略动作”：

1. 总监一眼能看见目标和四类机会。
2. 方案与实验能直接对应资源投入和验证节奏。
3. 页面整体更像战略简报，而不是 discovery 过程记录。
