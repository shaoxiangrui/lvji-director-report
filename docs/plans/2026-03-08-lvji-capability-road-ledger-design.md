# 驴迹总监汇报页 Capability Road Ledger Design

日期：2026-03-08

## 目标

优化 [lvji-director-report.html](/Users/wenjiayan/PycharmProjects/驴迹AI/lvji-director-report.html) 的 `P0 / 关键场景` 与 `P1 / 用户故事与执行` 两个区块。

本次调整包含四项：

1. 去掉 `Opportunity Solution Tree`。
2. 将 `stories-grid` 从 `P1 / 用户故事与执行` 移到 `P0 / 关键场景` 的 `roles-grid` 上方。
3. 去掉 `story-map` 和 `priority-grid`。
4. 在 `P1 / 用户故事与执行` 中新增“按能力划分的执行计划”。

## 设计判断

当前页面的主问题不是信息不够，而是 `P0` 和 `P1` 的证明链条不够顺。

具体体现在：

1. `OST` 更偏策略展开，会和三张关键场景卡争抢注意力。
2. `用户故事` 放在 `P1` 里，离 `关键场景` 太远，导致“场景 -> 角色 -> 具体故事”的阅读链被打断。
3. `story-map` 和 `priority-grid` 更像内部拆解材料，不适合总监汇报主页面。
4. `P1` 目前缺少一个真正回答“怎么落地”的模块。

因此，这次的核心不是加内容，而是重排证明顺序：

`关键场景 -> 角色 -> 用户故事 -> 执行计划`

这样更符合总监阅读逻辑。

## 结构调整

### 1. 去掉 Opportunity Solution Tree

处理方式：

1. 删除 `P0 / 关键场景` 中整块 `OST` 内容。
2. 不保留替代占位模块。
3. `P0` 只保留三张关键场景卡、角色价值条，以及移动后的用户故事。

原因：

1. 当前汇报主问题不是缺战略发散，而是缺高层可扫读的落地路径。
2. `OST` 更适合附录或工作坊材料，不适合继续占据主页面篇幅。

### 2. stories-grid 上移到 roles-grid 上方

处理方式：

1. 保持 `stories-grid` 的内容和样式完全不变。
2. 仅调整其所在位置。
3. 新顺序为：
   - 三张关键场景卡
   - `stories-grid`
   - `roles-grid`

原因：

1. 用户故事本质上是场景的可执行翻译。
2. 把它放到角色条上方，能更快建立“场景 -> 具体用户需求 -> 价值对象”的关系。

### 3. 去掉 story-map 和 priority-grid

处理方式：

1. 删除 `story-map`。
2. 删除 `priority-grid`。
3. 不再保留旧的版本切片和优先级卡。

原因：

1. 这两块更像交付团队内部推进材料。
2. 对总监而言，真正需要的是按能力划分的执行计划，而不是 backlog 展开形式。

## 新增模块：Capability Road Ledger

### 模块定位

这是 `P1 / 用户故事与执行` 的核心替代模块。

它要回答的问题不是“有哪些故事”，而是：

1. 先做哪项能力。
2. 每项能力在什么时间推进。
3. 要做哪些关键动作。
4. 最终交付什么。

### 展示方式

采用 `Capability Road Ledger` 风格。

这不是后台表格，也不是时间轴，而是一张高层执行台账。

### 视觉方向

保持当前 `Boardroom Atlas` 风格一致：

1. 继续使用米白纸面底色。
2. 继续使用墨绿作为主色。
3. 继续使用铜色标记优先级。
4. 保持当前卡片的纸面质感和轻阴影。

### 结构设计

每一行展示：

1. `优先级`
2. `能力`
3. `时间`
4. `关键动作`
5. `交付物`

展示原则：

1. 优先级单独做成显眼标签。
2. 能力名称作为每行主标题。
3. 时间信息紧跟能力展示，帮助总监快速感知节奏。
4. “关键动作”与“交付物”做成双列信息块，但保持纸面简报感。

### 内容范围

按用户提供的内容完整承载以下能力：

1. `P0 / 文旅知识中枢 v1`
2. `P0 / 数字交付员工 v1`
3. `P0 / 安全与治理`
4. `P1 / 数字导游员工 v1`
5. `P1 / 数字营销员工 v1`
6. `P1 / 商业化包装`
7. `P2 / 数字运营员工 v1`
8. `P2 / ROI 看板与续约策略`

## 页面顺序调整后

### P0 / 关键场景

1. 三张关键场景卡
2. `stories-grid`
3. `roles-grid`

### P1 / 用户故事与执行

1. `epic-grid`
2. `Capability Road Ledger`

## 风格约束

1. 不改 `stories-grid` 的内容。
2. 不改 `stories-grid` 的样式。
3. 不改现有页面的色彩系统。
4. 新增执行计划模块必须像董事会执行台账，而不是通用数据表。
5. 删除 `OST` 后不再新增新的战略图示型模块。

## 预期结果

调整后，整页的阅读链条会更顺：

1. `P0` 先讲场景，再讲用户故事。
2. `P1` 不再讲拆解细节，而直接讲按能力划分的执行计划。
3. 页面整体会更接近“总监汇报”，而不是“产品工作坊记录”。
