## 0. 方案定位

这份方案聚焦你未来一年工作计划里的第一个基础工程：`文旅知识中枢`。  
目标不是做一个“知识库页面”，而是做一套能被数字交付员工、数字导游员工、数字营销员工、数字运营员工稳定调用的 `可治理知识底座`。

本方案按你指定的技能链组织：

1. `prompt-engineering`：把任务改写成标准 prompt
2. `recommendation-canvas + jobs-to-be-done + positioning-statement + problem-statement + epic-hypothesis`：输出产品设计方案
3. `knowledge-graph-construction`：输出可执行的数据与图谱建设方案

## 步骤1：标准 Prompt

下面这段 prompt 可以直接给 AI 助手、外部顾问、内部团队协作用。

```text
你是一名服务于驴迹科技集团有限公司的资深 AI 产品与知识中枢架构顾问。

<背景>
驴迹科技集团有限公司的年度 AI 战略已经明确：未来 12 个月优先建设“1 个文旅 AI 中枢 + 4 类数字员工”，其中 P0 的第一优先级是知识中枢和数字交付员工。公司已有旅游内容、讲解词、GIS/地图、项目案例、SOP、客户网络和文旅产品入口，但这些资产分散在不同系统、项目和文档中，尚未形成统一、可治理、可追溯、可被 AI 调用的上下文系统。
</背景>

<任务>
请为“驴迹文旅知识中枢”输出一份完整建设方案，目标是支撑数字交付员工 v1，并为后续数字导游员工、数字营销员工和数字运营员工提供统一底座。
</任务>

<必须回答的问题>
1. 如何确定知识中枢的领域范围和数据源边界
2. 每类数据源如何获取，采用什么接入方式
3. 数据如何做解析、清洗、标准化、去重、权限分类、版本管理和入库
4. 知识图谱或知识中枢的数据模型如何设计
5. 入库指标、质量门槛、验收标准如何设定
6. 建设过程的详细执行步骤、角色分工、时间计划和交付物
</必须回答的问题>

<约束>
1. 优先支持 P0 场景，不做无限扩张

5. 不要把方案写成泛化的数据中台方案，必须贴合文旅行业和驴迹现有资产
</约束>

<建议方法>

</建议方法>

<输出格式>
A. 一页结论
B. 产品设计方案
C. 知识中枢架构与数据模型
D. 数据源范围与获取方案
E. 数据清洗与入库流程
F. 入库指标和验收门槛
G. 12 周执行计划
H. 风险与边界
</输出格式>

<输出要求>
1. 用中文输出
2. 结论优先，结构清晰
3. 尽量表格化，但避免空泛
4. 所有建议都要能落到执行动作
</输出要求>
```

## 步骤2：知识中枢产品设计方案

## 2.1 一页结论

### 结论

驴迹的知识中枢不应被定义为“公司资料库”，而应被定义为：

**面向文旅场景、可治理、可追溯、可权限控制的知识操作系统。**

它的第一阶段目标不是覆盖所有知识，而是先解决两个最值钱的问题：

1. 让 `数字交付员工 v1` 能稳定完成案例检索、方案生成、FAQ 回答、培训材料生成
2. 让后续 `数字导游员工` 能调用可信的景区、路线、设施、演出、注意事项等标准知识

### P0 范围判断

`P0 必须纳入`

- 景区主数据与空间点位
- 导览内容与讲解素材
- 驴迹产品资料与方案模板
- 历史项目案例、FAQ、SOP、培训材料
- 知识来源、版本、权限、更新时间

`P0 暂不纳入`

- 实时客流调度、IoT 告警、复杂运营决策
- 财务/行政/HR 等通用办公知识
- 没有 owner 的历史文档堆
- 无授权的第三方旅游内容抓取

### 产品目标

90 天内交付一个 `知识中枢 v1`，满足：

1. 交付团队可以按景区、客户、产品、项目阶段快速检索知识
2. AI 回答和生成内容必须带引用来源
3. 每条知识必须知道 `谁提供、何时更新、适用于谁`
4. 不同客户、不同租户知识严格隔离

## 2.2 Recommendation Canvas

### Business Outcome

| 项目 | 目标 |
|---|---|
| 内部效率 | 降低售前和交付团队找资料、拼方案、做培训包的工时 |
| 毛利改善 | 提升历史知识复用率，减少重复劳动和返工 |
| 对外能力 | 为数字导游、数字营销、数字运营提供统一底座，避免后续重复建设 |
| 组织杠杆 | 把个人经验沉淀为组织资产，而不是继续依赖少数老员工 |

### Product Outcome

| 用户 | 结果 |
|---|---|
| 交付经理 | 需求一进来，就能找到相似案例、产品模块、SOP 和方案结构 |
| 内容运营/知识管理员 | 知道哪些知识有效、哪些过期、哪些冲突、哪些需要更新 |
| AI 应用调用方 | 能拿到可引用、可审计、可控制权限的知识，而不是无源回答 |

### 核心问题

驴迹已经拥有大量文旅内容、项目经验、GIS 资产和产品资料，但这些内容分散在导览内容库、项目文档、历史方案、运营资料和个人经验里，没有统一主数据、统一权限和统一引用机制，导致：

1. 交付知识无法高效复用
2. AI 无法获得稳定上下文
3. 组织无法判断哪些内容可信、是否过期、适用于哪个客户

### 解决方案假设

如果驴迹先建设一套可治理的文旅知识中枢，把 `景区主数据 + 导览内容 + 项目案例 + SOP + 产品资料 + 权限/版本` 统一起来，并先用在数字交付员工上，那么公司会先拿到内部提效和毛利改善，再为后续对外 AI 模块提供稳定底座。

### 假设与未知

| 类别 | 内容 |
|---|---|
| 假设 | 历史项目文档中有足够多可复用内容，值得结构化沉淀 |
| 假设 | 业务团队愿意配合定义知识 owner、更新时间和仲裁规则 |
| 假设 | 先服务数字交付员工，可以最快形成 ROI 和组织 adoption |
| 未知 | 历史文档质量参差程度是否足以支持批量解析 |
| 未知 | 各业务线是否接受统一主数据口径 |
| 未知 | 不同客户项目里的私域知识隔离复杂度有多高 |

### 风险

| 风险 | 表现 | 缓释 |
|---|---|---|
| 数据口径冲突 | 同一景区、同一产品、同一指标有多个版本 | 建主数据字典、来源优先级、口径仲裁机制 |
| 文档不可用 | 扫描件、图片版 PDF、缺失字段多 | 先做样本抽检，低质量文档不纳入 P0 |
| 权限泄露 | 客户私域资料被错误召回 | 租户分域、强制权限标签、审计日志 |
| 组织不维护 | 知识上线后无人更新 | 每个知识域明确 owner 和更新 SLA |
| 范围失控 | 把所有历史资料都拉进来 | 只纳入 P0 场景直接需要的数据源 |

### 成功指标

| 维度 | 指标目标 |
|---|---|
| 知识覆盖 | P0 必需数据源接入率 >= 90% |
| 可用性 | 检索命中率 >= 75%，回答引用率 >= 85% |
| 质量 | 关键字段完整率 >= 90%，重复率 <= 5% |
| 治理 | 来源字段覆盖率 = 100%，权限标签覆盖率 = 100% |
| 业务 | 方案初稿产出时间下降 >= 50%，文档复用率 >= 60% |

### What's Next

1. 先锁定 P0 使用场景和数据范围
2. 再做数据源清单和接入合同
3. 再做解析、清洗、图谱和混合检索
4. 先用数字交付员工验证价值，再扩游客问答

## 2.3 Jobs-to-be-Done

## 用户1：驴迹交付经理

### Functional Jobs

1. 快速找到相似景区、相似客户、相似项目的历史案例
2. 生成方案初稿、FAQ、培训材料和交付清单
3. 判断哪些内容可复用，哪些内容必须重写

### Social Jobs

1. 在客户面前显得专业、反应快、有行业沉淀
2. 在内部显得不是“只会找人要资料”，而是能独立推进项目

### Emotional Jobs

1. 减少因为资料找不到、内容口径不一致带来的焦虑
2. 对方案内容是否准确更有把握

### Pains

1. 历史文档分散在个人电脑、项目群、网盘和旧系统里
2. 同一景区信息有多个版本，不知道哪个最新
3. 老员工经验没有结构化沉淀，新人很难复用

### Gains

1. 5 分钟内找到可用案例
2. 一键生成带引用的方案框架
3. 能看到内容是否过期、来自哪里、适用于哪些客户

## 用户2：知识管理员/内容运营

### Functional Jobs

1. 把导览内容、景区资料、SOP、案例持续更新到统一中枢
2. 处理冲突数据、过期数据、错误知识和新增知识
3. 管理不同租户、不同客户、不同业务线的权限边界

### Social Jobs

1. 被视为知识 owner，而不是被动“资料管理员”

### Emotional Jobs

1. 知道自己的工作在直接提升 AI 准确率，而不是机械搬运

### Pains

1. 现在没有统一的知识结构和入库标准
2. 上传完资料后，不知道是否被真正调用
3. 错误知识被用了以后，追溯成本很高

### Gains

1. 每条知识都有 owner、版本、更新时间和引用次数
2. 能知道哪些知识高频被调用，优先维护哪些内容

## 用户3：AI 应用/数字员工调用方

### Functional Jobs

1. 获取可信知识而不是泛化回答
2. 在回答、生成、推荐时保留出处和权限判断
3. 在不确定时把问题交回人工审批

### Pains

1. 当前没有统一可治理上下文，问答质量依赖临时拼接
2. 没有图谱关系时，多跳查询和口径统一很难做

### Gains

1. 有标准化的实体、关系、引用和版本字段可调用
2. 能根据租户和场景限制召回范围

## 2.4 Problem Statement

### 问题叙事

**我是一名** 驴迹交付经理/知识管理员，  
**我正在尝试** 用最短时间把景区知识、项目案例、产品资料和 SOP 组织起来，支持方案编写、客户答疑和 AI 助手调用，  
**但现在** 这些知识分散在不同系统、历史项目、文档和个人经验里，版本不清、权限不清、更新不清，  
**因为** 公司还没有形成统一的文旅知识主数据、知识结构、接入标准和治理流程，  
**这让我感到** 每次都像重新开始，效率低，也不敢完全信任 AI 的输出。

### 最终问题陈述

驴迹需要一套统一、可追溯、可权限控制的文旅知识中枢，让内部团队和数字员工能够基于可信知识完成检索、生成、引用和协同，否则 AI 只能停留在点状演示，无法稳定支撑交付效率和后续产品化。

---

## 2.5 Positioning Statement

### Value Proposition

**For** 驴迹的售前、交付、内容运营和后续数字员工应用  
**that need** 把分散的景区内容、地图、项目案例、SOP 和产品知识统一成可信、可调用的业务上下文  
**驴迹文旅知识中枢**  
**is a** 面向文旅场景的知识操作系统  
**that** 提供标准主数据、来源引用、权限控制、版本管理和图谱化关系，支撑检索、生成、审批和后续数字员工能力

### Differentiation Statement

**Unlike** 通用知识库或泛办公 AI 助手  
**驴迹文旅知识中枢**  
**provides** 基于景区内容、GIS、项目经验和文旅业务流程的行业上下文，既能支撑内部提效，也能沉淀为对外可售卖能力的底座

## 2.6 Epic Hypothesis

| Epic | 假设 | Tiny Acts of Discovery | 成功标准 |
|---|---|---|---|
| Epic 1：领域与主数据治理 | 如果我们先定义景区、POI、路线、产品、项目、SOP 等核心实体及统一口径，那么后续所有知识接入和 AI 调用都会稳定很多 | 选 2 个景区、2 个项目做主数据映射试点 | 主数据映射准确率 >= 95%，重复命名冲突下降 >= 80% |
| Epic 2：多源数据接入与清洗 | 如果我们把 PDF、PPT、Word、Excel、结构化表统一进入标准解析清洗管道，那么可复用知识会从“找不到”变成“可搜索、可引用” | 抽 500 份历史资料做解析试点 | 解析成功率 >= 95%，关键字段完整率 >= 90% |
| Epic 3：图谱检索与引用 | 如果我们把实体关系和文档片段同时建索引，并强制回答带来源，那么 AI 回答会更可信 | 构建 100 个 benchmark query 做离线评测 | Top5 检索命中率 >= 80%，回答引用率 >= 85% |
| Epic 4：权限、版本和反馈闭环 | 如果每条知识都有租户、来源、版本、审批和反馈回写，那么知识中枢可以长期运营，而不是一次性项目 | 在 1 个内部团队试运行反馈闭环 | 权限标签覆盖率 = 100%，错误知识闭环时间 <= 3 个工作日 |
| Epic 5：数字交付员工首个消费场景 | 如果先让知识中枢支撑方案生成、FAQ 回答和培训材料生成，那么组织会更快看到 ROI | 让交付团队连续试用 4 周 | 方案初稿时间下降 >= 50%，试点周活 >= 70% |

## 2.7 产品边界

### P0 做到什么程度

1. 支持 `景区/项目/产品/SOP/FAQ` 的统一检索
2. 支持文档级和片段级引用
3. 支持租户、客户、内部资料的权限分层
4. 支持主数据、版本、更新时间、知识 owner
5. 支持数字交付员工优先调用

### P0 不承诺什么

1. 不承诺一次性把所有历史知识都清完
2. 不承诺没有人工审核也能自动产出对外正式材料
3. 不承诺 P0 就覆盖实时客流、报警、调度等复杂运营数据
4. 不承诺图谱一上就能替代所有搜索和 BI

## 步骤3：Knowledge Graph Construction 可执行方案

## 3.1 建设原则

1. `从场景倒推数据`，不从“全量收集”开始
2. `先主数据，再文档`，不先把脏文档全灌进去
3. `先治理，再开放调用`，每条知识必须带来源、版本、权限
4. `图谱 + 文档检索混合`，不是只做向量库，也不是只做关系库
5. `先支持数字交付员工`，再扩数字导游/营销/运营

---

## 3.2 Step 1：如何确定数据源和领域范围

### 3.2.1 先锁定 P0 业务问题

知识中枢不是抽象平台，必须先回答这 5 个 P0 问题：

1. 某个客户需求进来后，最相似的历史项目和方案是什么
2. 某个景区的介绍、点位、路线、设施、FAQ、注意事项是什么
3. 某个产品模块能解决什么问题，适配哪些客户场景
4. 某类交付场景对应的 SOP、培训包、验收要点是什么
5. 某条知识是不是最新、是谁维护的、能不能给当前租户使用

### 3.2.2 再从问题反推知识域

P0 推荐只建 5 个知识域：

| 知识域 | 作用 | P0 是否必须 |
|---|---|---|
| 景区主数据域 | 统一景区、景点、POI、设施、路线等实体 | 必须 |
| 导览内容域 | 讲解词、景区介绍、游览建议、注意事项、FAQ | 必须 |
| 产品与方案域 | 产品模块、方案模板、投标材料、能力说明 | 必须 |
| 项目与交付域 | 历史案例、实施 SOP、培训材料、验收标准 | 必须 |
| 治理元数据域 | 来源、版本、owner、权限、更新时间、审批状态 | 必须 |

P1/P2 再扩的域：

- 运营指标域
- 营销素材与活动域
- 工单与客诉域
- 实时事件域

### 3.2.3 数据源确定方法

数据源不是靠感觉选，而是按下面 6 步判定：

#### 第 1 步：列候选源

把所有可能进入知识中枢的源先列出来，不做判断。

来源类型至少包含：

- 结构化库表
- 内部 API
- 文档库/NAS/网盘/对象存储
- 历史项目归档
- 官网/小程序/导览 CMS
- 官方外部公开资料

#### 第 2 步：补齐源信息

每个数据源必须填完整这 10 个字段：

| 字段 | 说明 |
|---|---|
| source_id | 数据源唯一编号 |
| source_name | 数据源名称 |
| domain | 所属知识域 |
| owner | 业务 owner |
| format | DB/API/PDF/Word/PPT/Excel/HTML 等 |
| update_freq | 更新频率 |
| tenant_scope | 公共/内部/客户私域 |
| sensitivity | 公开/内部/敏感 |
| access_method | API/SQL/导出/人工上传 |
| p0_relevance | 是否直接支持 P0 问题 |

#### 第 3 步：按 5 个维度评分

| 维度 | 评分解释 |
|---|---|
| 权威性 | 是否为官方、签字确认或公司正式 owner 数据 |
| P0 相关性 | 是否直接支撑 P0 场景 |
| 结构可解析性 | 是否便于解析和标准化 |
| 更新可持续性 | 是否能持续获取，而不是一次性拉取 |
| 权限清晰度 | 是否清楚知道哪些人可见 |

评分规则：

- 每项 1-5 分
- 总分 >= 20：纳入 P0
- 总分 16-19：放入 P1 备选
- 总分 <= 15：暂不纳入

#### 第 4 步：按“来源优先级”定主源

同一事实可能有多个来源，必须定主源。

推荐优先级：

1. 客户签字确认/项目验收资料
2. 驴迹正式发布和已审核内容
3. 景区官网/文旅局官方资料
4. 有授权的第三方数据
5. 非官方互联网内容

`P0 规则`：第 5 类不进知识中枢。

#### 第 5 步：按“知识单元”判断是否入库

不是所有文档都要整篇入库。  
你要判断的是 `哪一个知识单元` 值得入。

建议最小知识单元：

- 一个景区实体
- 一个 POI 点位
- 一条路线
- 一段讲解词
- 一个 FAQ 问答对
- 一个 SOP 步骤
- 一个项目案例摘要
- 一段方案模块说明

#### 第 6 步：定义首批必接入源

P0 首批推荐接入：

| 优先级 | 数据源 | 说明 |
|---|---|---|
| P0 | 景区主数据表/景区 CMS | 景区、景点、POI、设施、路线等基础实体 |
| P0 | 导览内容库 | 讲解词、介绍、FAQ、注意事项 |
| P0 | 产品资料库 | 电子导览、SaaS、智慧景区等模块资料 |
| P0 | 历史项目方案包 | 方案模板、项目案例、投标材料 |
| P0 | 交付 SOP/培训资料 | 实施流程、培训模板、验收标准 |
| P0 | 权限与客户台账 | 客户、租户、项目归属关系 |

### 3.2.4 推荐的首版领域边界

#### 首版必须回答的问题

1. 这个景区的标准资料是什么
2. 这个客户适合哪些产品模块
3. 有没有类似项目案例
4. 这个交付动作该参考哪些 SOP
5. 当前回答引用的内容来自哪里、属于哪个版本

#### 首版不做的问题

1. 节假日实时客流调度建议
2. 设备报警根因分析
3. 全量舆情和社媒抓取
4. 通用办公知识问答

## 3.3 Step 2：图谱数据模型选择

### 推荐结论

`P0 推荐采用 LPG（属性图）+ 文档向量检索 + 关系型元数据` 的混合架构。

### 选择理由

| 方案 | 是否推荐 | 原因 |
|---|---|---|
| 纯向量知识库 | 不推荐 | 长文检索快，但主数据、权限、关系和多跳查询弱 |
| 纯 RDF/OWL | 暂不推荐 P0 | 标准化强，但首期建模和团队门槛高，不利于快速落地 |
| LPG 属性图 | 推荐 | 适合企业知识、关系灵活、适合快速迭代和图谱+RAG 混合 |

### 推荐架构

| 层 | 推荐职责 |
|---|---|
| Raw 层 | 保留原始文件和原始表快照 |
| Clean 层 | 清洗后的结构化主数据和文档片段 |
| Graph 层 | 存实体、关系、来源、权限、版本 |
| Search 层 | 存全文索引和向量索引 |
| Service 层 | 给数字交付员工/数字导游员工提供检索、引用和权限判断 API |

## 3.4 Step 3：Schema 和 Ontology 设计

## 核心节点类型

| 节点类型 | 说明 | 核心属性 |
|---|---|---|
| ScenicArea | 景区 | scenic_id, name, alias, level, region, status |
| POI | 景点/设施点位 | poi_id, name, type, lat, lng, open_hours |
| Route | 路线 | route_id, name, duration, crowd_level |
| ContentAsset | 内容资产 | asset_id, title, content_type, language, version |
| FAQ | 问答对 | faq_id, question, answer, intent, version |
| ProductModule | 产品模块 | product_id, name, scenario, edition |
| ProjectCase | 项目案例 | project_id, customer_name, project_type, summary, tenant_id |
| SOP | 流程规范 | sop_id, name, step_no, owner_role |
| SourceDocument | 来源文档 | doc_id, source_id, file_path, author, updated_at |
| Tenant | 租户 | tenant_id, tenant_name, scope |
| KnowledgeChunk | 最小知识片段 | chunk_id, text, chunk_type, source_doc_id |
| UserRole | 角色 | role_id, role_name, access_level |

## 核心关系类型

| 关系 | 含义 |
|---|---|
| ScenicArea - HAS_POI -> POI | 景区包含点位 |
| ScenicArea - HAS_ROUTE -> Route | 景区包含路线 |
| ContentAsset - DESCRIBES -> ScenicArea/POI/Route | 内容描述某个实体 |
| FAQ - ANSWERS -> ScenicArea/ProductModule/SOP | FAQ 回答某类对象 |
| ProjectCase - USES_PRODUCT -> ProductModule | 某项目使用了哪些产品 |
| ProjectCase - FOLLOWS_SOP -> SOP | 某项目参考哪些 SOP |
| KnowledgeChunk - DERIVED_FROM -> SourceDocument | 片段来源于某文档 |
| KnowledgeChunk - BELONGS_TO -> Tenant | 片段属于哪个租户 |
| UserRole - CAN_ACCESS -> Tenant/Chunk | 角色可访问范围 |
| SourceDocument - HAS_VERSION -> ContentAsset/FAQ/SOP | 文档与知识版本关联 |

## 关键建模规则

1. `景区、POI、路线、产品、项目` 必须有唯一主键
2. `票价、开放时间、活动安排` 这类会变化的信息按 `时态事实` 存，必须带生效时间
3. `坐标` 统一维护主坐标体系，展示时再做地图坐标转换
4. `知识片段` 必须能回溯到原始文档和原始位置
5. `权限` 不是后补字段，而是入库必填字段

## 3.5 Step 4：如何获取数据

## 3.5.1 数据获取总体流程

1. 建立数据源台账
2. 为每个源签数据接入责任人
3. 明确接入方式
4. 先做小样本打通
5. 评估质量后再批量导入

## 3.5.2 按来源类型设计获取方式

### A. 结构化数据

适用：

- 景区主数据表
- 产品表
- 客户与项目台账
- FAQ 标签表

获取方式：

1. 优先走只读库或只读 API
2. 没有 API 时，用定时导出 CSV/Excel
3. 所有拉取任务必须保留批次号和时间戳

执行步骤：

1. 和系统 owner 确认表结构和字段含义
2. 输出字段字典
3. 标记必填字段和废弃字段
4. 建增量拉取规则
5. 先拉近 90 天数据做试导

### B. 半结构化文档

适用：

- 方案 PPT
- Word 方案
- 培训资料
- 投标书
- SOP 文档
- 验收材料

获取方式：

1. 从项目归档盘、NAS、对象存储批量收集
2. 每个文件先生成 `file_hash`
3. 按客户、项目、日期、文档类型补元数据

执行步骤：

1. 先取近 2 年高复用项目资料
2. 按模板类、案例类、SOP 类分批
3. 对扫描件单独标记 OCR 路径
4. 对无法识别或无 owner 文档先隔离

### C. 导览内容和内容资产

适用：

- 讲解词
- 图文导览
- 景区介绍
- 游览建议
- FAQ

获取方式：

1. 优先从现有内容 CMS 或内容库导出
2. 如果内容在多个系统，先统一成内容资产台账
3. 每条内容都要带语言、适用景区、版本、审核状态

执行步骤：

1. 抽样检查内容库是否存在重复版本
2. 把一篇长内容切成可引用片段
3. 把片段挂到景区/POI/路线实体上

### D. 外部官方资料

适用：

- 景区官网说明
- 文旅局公告
- 官方票务/演出安排

获取方式：

1. 只接官方源或已授权源
2. 先人工确认采集合法性与版权边界
3. 对高变化信息必须带更新时间和抓取时间

执行步骤：

1. 建立外部源白名单
2. 每个外部源设 owner
3. 只补内部缺失字段，不覆盖内部确认过的数据

## 3.6 Step 5：数据怎么清洗

数据清洗不是一步，而是一条标准流水线。

## 3.6.1 清洗总流程

1. `原始落盘`
2. `解析抽取`
3. `文档分类`
4. `字段标准化`
5. `主数据匹配`
6. `去重合并`
7. `权限和敏感信息处理`
8. `知识切片`
9. `实体关系抽取`
10. `质量校验`
11. `人工抽检`
12. `发布入库`

## 3.6.2 每一步怎么做

### 第 1 步：原始落盘

动作：

1. 所有源先进入 Raw 层
2. 保留原始文件、原始表、原始 API 响应
3. 生成 batch_id、source_id、ingested_at

要求：

1. Raw 层不做覆盖写
2. 每次导入都可回放

### 第 2 步：解析抽取

动作：

1. PDF/Word/PPT/Excel 分别走专用解析器
2. 扫描件走 OCR
3. 表格内容和正文内容分别抽取

输出：

- 纯文本
- 表格
- 页面/段落定位信息
- 解析状态码

### 第 3 步：文档分类

将文档分成：

1. 景区资料
2. 导览内容
3. 方案模板
4. 历史案例
5. SOP/培训材料
6. FAQ
7. 不可用文档

规则：

- 不可用文档进入人工复核队列

### 第 4 步：字段标准化

对关键字段统一格式：

| 字段 | 标准化规则 |
|---|---|
| 景区名称 | 统一正式名 + 别名词典 |
| 地区 | 统一省/市/区县编码 |
| 坐标 | 统一数值格式和坐标系标识 |
| 时间 | 统一到 `YYYY-MM-DD HH:mm:ss` |
| 产品名 | 统一为标准产品目录名 |
| 项目阶段 | 统一为售前/实施/培训/验收/运维 |
| 文档日期 | 缺失时按文件元数据补默认值并标低可信 |

### 第 5 步：主数据匹配

动作：

1. 用字典匹配 + 模糊匹配 + 人工确认三段式做实体归一
2. 建立别名表
3. 生成 canonical_id

例子：

- “广州白云山”
- “白云山风景区”
- “白云山景区”

归并为同一个 `scenic_id`

### 第 6 步：去重合并

去重原则：

1. 同 hash 文档直接去重
2. 同标题 + 同客户 + 相近日期 + 高相似文本，标记为候选重复
3. FAQ 按问题意图和答案相似度去重

冲突处理：

1. 不自动覆盖高权威来源
2. 把冲突版本都保留
3. 由知识 owner 做仲裁

### 第 7 步：权限和敏感信息处理

所有数据必须打上：

1. tenant_id
2. visibility_level
3. sensitivity_level
4. approval_status

敏感内容处理：

1. 客户合同金额、联系人、手机号等字段默认脱敏
2. 涉及未公开客户信息的内容不能进入公共知识域

### 第 8 步：知识切片

不要把一整份文档当作一个知识单元。

切片规则建议：

| 类型 | 切片方式 |
|---|---|
| 景区介绍 | 按主题段切 |
| 讲解词 | 按景点或讲解段切 |
| SOP | 按步骤切 |
| 项目案例 | 按背景/方案/结果切 |
| FAQ | 一问一答切 |

每个 chunk 必须保存：

- chunk_id
- source_doc_id
- source_location
- entity_refs
- tenant_id
- version

### 第 9 步：实体关系抽取

抽取目标不是“抽一切”，而是抽支持业务问题的关系。

P0 必抽关系：

1. 景区包含哪些 POI
2. 景区有哪些路线
3. 内容描述了哪个景区/点位/路线
4. 项目用了哪些产品模块
5. 项目遵循哪些 SOP
6. FAQ 适用于哪个对象

抽取方法：

1. 规则优先处理强结构关系
2. LLM 辅助处理弱结构文本关系
3. 关系写入前做 schema 校验

### 第 10 步：质量校验

自动校验至少覆盖：

1. 必填字段是否为空
2. 权限字段是否为空
3. source_id/version 是否为空
4. 主实体是否存在孤儿数据
5. 坐标、时间、编码格式是否正确
6. chunk 是否关联来源文档

### 第 11 步：人工抽检

抽检样本：

1. 每批次随机抽 5%-10%
2. 高风险文档全检
3. 新接入源首批必须全流程抽检

抽检重点：

1. 实体是否归并正确
2. 关系是否合理
3. 引用是否能回到原文
4. 权限是否正确

### 第 12 步：发布入库

通过门禁后才允许进 Serving 层。

入库对象：

1. 图谱节点与关系
2. 文本索引
3. 向量索引
4. 元数据表

## 3.7 Step 6：入库指标如何设定

入库指标分成 5 层：`源接入、清洗、图谱、检索、治理`。

## 3.7.1 源接入指标

| 指标 | 定义 | P0 门槛 |
|---|---|---|
| 源接入完成率 | 已接入 P0 数据源 / 计划 P0 数据源 | >= 90% |
| 增量同步成功率 | 成功执行的增量任务 / 总增量任务 | >= 98% |
| 首批源 owner 完整率 | 已定义 owner 的源 / 已接入源 | = 100% |

## 3.7.2 清洗指标

| 指标 | 定义 | P0 门槛 |
|---|---|---|
| 解析成功率 | 成功解析文档数 / 总文档数 | >= 95% |
| 必填字段完整率 | 必填字段完整记录 / 总记录 | >= 90% |
| 重复率 | 被识别为重复的有效记录 / 总记录 | <= 5% |
| 主数据归并准确率 | 抽检中归并正确样本 / 归并样本 | >= 95% |
| 敏感信息识别命中率 | 命中脱敏规则样本 / 应命中样本 | >= 95% |

## 3.7.3 图谱入库指标

| 指标 | 定义 | P0 门槛 |
|---|---|---|
| 实体抽取准确率 | 抽检正确实体 / 抽检实体 | >= 92% |
| 关系抽取准确率 | 抽检正确关系 / 抽检关系 | >= 85% |
| 孤儿节点率 | 无有效关系且不应独立存在的节点 / 总节点 | <= 3% |
| 来源追溯完整率 | 带 source_id、chunk_id、version 的知识 / 总知识 | = 100% |

## 3.7.4 检索与回答指标

| 指标 | 定义 | P0 门槛 |
|---|---|---|
| Benchmark Top5 命中率 | Top5 内至少出现 1 个正确片段的 query / 总 query | >= 80% |
| 回答引用率 | 带来源引用回答 / 总回答 | >= 85% |
| 无法回答转人工率 | 被正确拒答或转人工的问题 / 高风险问题 | >= 95% |
| 答案人工可接受率 | 人工评测可接受答案 / 抽检答案 | >= 85% |

## 3.7.5 治理指标

| 指标 | 定义 | P0 门槛 |
|---|---|---|
| 权限标签覆盖率 | 具备 tenant 和 visibility 的知识 / 总知识 | = 100% |
| 更新时间覆盖率 | 具备 updated_at 的知识 / 总知识 | >= 95% |
| SLA 达标率 | 在规定时限内更新的知识 / 需更新知识 | >= 90% |
| 错误知识闭环时间 | 从发现到修复的中位时间 | <= 3 个工作日 |
| 权限泄露缺陷数 | 越权召回问题数 | = 0 |

### 入库门禁

任一批次出现以下情况，不允许发布到 Serving 层：

1. 来源追溯完整率 < 100%
2. 权限标签覆盖率 < 100%
3. 关键字段完整率 < 90%
4. 新源未做人工抽检
5. benchmark 命中率未达标

## 3.8 详细执行步骤

下面按 `12 周` 给出执行方案。

## 第 1-2 周：锁场景、定范围、建台账

### 目标

把“要做什么知识”先说清，不直接开接数。

### 动作

1. 确认 P0 只服务 `数字交付员工 v1 + 导览问答基础`
2. 输出 P0 业务问题清单
3. 建数据源台账模板
4. 组织景区内容、交付、产品、技术四方盘点数据源
5. 明确每个知识域的业务 owner

### 交付物

- P0 业务问题清单
- 知识域清单
- 数据源台账 v1
- owner 名单

### 退出标准

1. 每个知识域都有人负责
2. 所有候选源完成评分

## 第 3-4 周：定主数据和接入合同

### 目标

先把主数据口径定住。

### 动作

1. 定义景区、POI、路线、产品、项目、SOP 主数据字典
2. 输出字段字典和取值规则
3. 选 6 个高优先级源进入首批接入
4. 为每个源确定接入方式和更新频率
5. 明确来源优先级和冲突仲裁规则

### 交付物

- 主数据字典 v1
- 字段标准 v1
- 数据接入合同 v1
- 冲突仲裁规则

### 退出标准

1. 核心实体都有 canonical_id 规则
2. 6 个 P0 数据源的 access path 已打通

## 第 5-6 周：跑通解析和清洗流水线

### 目标

让结构化和文档类数据都能进入 Clean 层。

### 动作

1. 打通结构化源增量同步
2. 打通文档解析和 OCR
3. 实现文档分类
4. 建立标准化、去重、脱敏规则
5. 建知识切片规则

### 交付物

- Raw/Clean pipeline v1
- 文档分类规则
- 去重规则
- 脱敏规则
- 切片规则

### 退出标准

1. 解析成功率 >= 95%
2. 样本文档能稳定切出可引用 chunk

## 第 7-8 周：建图谱和混合检索

### 目标

把清洗后的数据变成可查询的知识结构。

### 动作

1. 创建核心节点和关系模型

## 3.10 风险与边界

### 高风险点

1. `历史文档质量差`
   处理：先做样本审计，不合格源不进 P0

2. `业务 owner 不清`
   处理：没有 owner 的知识域不开放为正式知识

3. `客户私域数据混用`
   处理：强制 tenant_id 和 visibility 字段，发布前做越权测试

4. `把知识中枢做成大而全平台`
   处理：P0 只围绕数字交付员工和导览基础能力

5. `只建库不运营`
   处理：把更新 SLA、错误闭环、引用率都纳入治理指标

### 明确边界

1. 知识中枢是 `组织底座`，不是一个单独销售页面
2. 知识中枢要先支撑最值钱的业务动作，不先追全量
3. 对外高风险内容必须保留人工审批
4. 没有来源和权限的数据，不允许进入 Serving 层

## 3.11 给 AI 产品经理的执行提醒

你作为 AI 产品经理，推进知识中枢时最容易踩的坑有三个：

1. `从技术栈开始，而不是从 P0 问题开始`
2. `把所有历史资料都当资产，而不是先判断哪些值得治理`
3. `把上线当终点，而不是把治理和 owner 机制当终点`

你第一阶段最重要的不是证明“模型多强”，而是证明：

1. 驴迹有一套可持续更新的文旅知识标准
2. AI 调用这套知识后，确实让交付更快、更稳
3. 这套底座未来能自然扩到导游、营销、运营，而不是推倒重来

## 3.12 最终建议

本年度知识中枢建设应坚持下面的顺序：

1. `先锁 P0 场景`
2. `先定知识域和主数据`
3. `先打通 6 个高价值源`
4. `先做来源、权限、版本和引用`
5. `先接数字交付员工验证 ROI`
6. `再扩到游客导览和营销/运营`

一句话收口：

**驴迹的知识中枢，不是为了“把资料集中起来”，而是为了把文旅知识变成数字员工可以稳定调用的生产资料。**
