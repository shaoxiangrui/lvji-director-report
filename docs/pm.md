| acquisition-channel-advisor | 用单位经济、客户质量和可扩展性评估获客渠道，并给出 scale/test/kill 决策。 | 比较内容营销、渠道代理、付费广告三种渠道，决定下季度预算分配。 |

| business-health-diagnostic | 从增长、留存、效率、资本健康四个维度诊断 SaaS 业务状态并排序问题优先级。 | 季度经营复盘时输出业务健康评分和紧急修复清单。 |
| company-research | 基于公开信息提取公司高管观点、产品策略和组织动态，形成战略情报画像。 | 面试前快速建立目标公司的产品战略与组织认知。 |
| context-engineering-advisor | 诊断“上下文堆砌”问题，设计更有效的记忆结构、检索策略与执行循环。 | 优化 AI 助手提示词与知识库调用方式，减少无效上下文。 |
| customer-journey-map | 构建从认知到忠诚的完整客户旅程图，覆盖动作、触点、情绪、KPI 与团队责任。 | 梳理 B2B SaaS 从试用到续费的全流程体验并找出断点。 |
| customer-journey-mapping-workshop | 通过引导式提问组织客户旅程地图工作坊，沉淀跨团队共识与改进机会。 | 组织产品、运营、客服共创 onboarding 旅程并统一问题定义。 |
| director-readiness-advisor | 面向 PM/新总监的角色跃迁辅导，提供情境化的 Director 能力建议。 | 准备 Director 面试时评估管理、组织协同与战略能力缺口。 |
| discovery-interview-prep | 规划客户发现访谈目标、样本和提纲，降低偏差并提高访谈有效性。 | 两周内要做 10 场访谈时，先产出访谈计划与问题脚本。 |
| discovery-process | 提供从问题假设到实验验证的端到端发现流程，避免“先做再验证”。 | 新功能立项前跑完整 discovery 流程确认是否值得投入开发。 |
| eol-message | 生成清晰且有同理心的下线/停服沟通文案，说明原因与迁移路径。 | 发布旧版功能下线通知并引导客户迁移到替代方案。 |
| epic-breakdown-advisor | 按 Humanizing Work 方法将 Epic 系统拆分成可独立交付的用户故事。 | 把“企业权限重构”拆成每两周可上线的价值切片。 |
| epic-hypothesis | 将 Epic 写成可验证的 if/then 假设，并定义成功指标与验证方式。 | 把“智能推荐系统”改写为假设并明确实验成功阈值。 |
| executive-onboarding-playbook | 提供 VP/CPO 30-60-90 天上岗剧本，强调先诊断再决策。 | 新任 VP 入职后安排访谈、组织诊断与节奏化对齐计划。 |
| feature-investment-advisor | 用收益、成本、ROI 与战略价值评估功能投资，输出 build/don’t build 建议。 | 评估“高级报表中心”是否值得投入一个季度开发资源。 |
| finance-based-pricing-advisor | 评估调价对 ARPU、转化、流失、NRR、回本周期的财务影响并给出 go/no-go。 | 决定是否将标准版价格从 99 提至 129，并预估流失风险。 |
| finance-metrics-quickref | 提供 32+ SaaS 财务指标速查（公式、基准、使用时机、红旗信号）。 | 经营会中快速查询 Magic Number 公式及合理区间。 |
| jobs-to-be-done | 用 JTBD 识别用户功能/情感/社会任务，发现真实需求与未满足收益。 | 分析用户为什么“雇佣”你的产品，而不是只收集功能请求。 |
| lean-ux-canvas | 引导使用 Lean UX Canvas 对齐业务问题、关键假设和学习计划。 | 在探索期组织 1 小时画布会，快速明确验证路径。 |
| opportunity-solution-tree | 构建 OST：从目标结果到机会空间、候选方案与验证实验。 | 将“老板要做 AI 功能”转成结构化机会树并定义首个实验。 |
| pestel-analysis | 系统分析政治、经济、社会、技术、环境、法律等外部因素对产品的影响。 | 评估进入海外市场时的监管风险与宏观机会。 |
| pol-probe | 设计 PoL（Proof of Life）轻量验证探针，以最小成本验证关键假设。 | 在开发前用假门页/手工服务验证需求强度。 |
| pol-probe-advisor | 根据假设类型、风险和资源约束，推荐最合适的 PoL 验证方式。 | 预算紧张时选择可在 1 周内执行的低成本验证方案。 |
| positioning-statement | 生成 Geoffrey Moore 风格定位陈述，明确目标用户、价值与差异化。 | 对齐产品、市场、销售三方的统一价值主张。 |
| positioning-workshop | 通过引导式工作坊共创产品定位，明确客群、需求、品类与竞争区隔。 | 和创始团队一起确定新产品首发定位与关键信息。 |
| prd-development | 结构化产出 PRD，整合问题定义、用户研究、方案与成功标准。 | 将分散在会议记录中的需求整理成可执行 PRD。 |
| press-release | 使用 Working Backwards 方法先写发布稿，先验证价值叙事再进入开发。 | 在立项前写“未来发布新闻稿”检验故事是否有吸引力。 |
| prioritization-advisor | 根据产品阶段与组织情境推荐优先级框架并给出落地方法。 | 在 RICE、Kano、MoSCoW 之间选型并排序季度需求池。 |
| problem-framing-canvas | 用 MITRE 问题框架（Look Inward/Outward/Reframe）澄清真正问题。 | 立项前先验证“我们要解决的问题”是否成立。 |
| problem-statement | 从用户视角写出清晰问题陈述，避免直接跳到方案。 | 把“做消息中心”改写为可验证的用户痛点陈述。 |
| product-strategy-session | 串联定位、问题框定、客户发现与路线图，形成端到端产品策略会。 | 半天战略工作坊产出年度产品方向与执行优先级。 |
| proto-persona | 快速生成假设型 Persona，指导早期设计决策并标注待验证项。 | 产品早期先产出首版用户画像，指导访谈与原型方向。 |
| recommendation-canvas | 用结构化画布评估 AI 产品方案的业务价值、风险与可行性。 | 向管理层提交 AI 功能立项建议与决策依据。 |
| roadmap-planning | 组织战略路线图规划，连接优先级、Epic、发布节奏与协同机制。 | 产出下一财年路线图并对齐研发、销售、运营节奏。 |
| saas-economics-efficiency-metrics | 分析 SaaS 单位经济与资本效率（CAC/LTV/回本/Rule of 40 等）。 | 融资前评估增长策略是否健康且可持续。 |
| saas-revenue-growth-metrics | 计算并解读收入与增长指标（MRR/ARR/Churn/NRR 等）。 | 月度经营会定位净收入留存下滑的核心原因。 |
| skill-authoring-workflow | 将原始 PM 方法论内容转成合规、可发布、可复用的 Skill。 | 把内部培训材料沉淀为团队可复用的 AI skill。 |
| storyboard | 生成 6 帧叙事故事板，把用户问题到解决路径可视化表达。 | 向高层演示新功能如何改善用户关键旅程。 |
| tam-sam-som-calculator | 引导计算 TAM/SAM/SOM，并形成有依据的市场规模估算。 | 在融资 BP 中完成市场规模章节与假设说明。 |
| user-story | 生成标准用户故事与 Gherkin 验收条件，提升研发与测试一致性。 | 将需求讨论结果转成可开发、可测试的 stories。 |
| user-story-mapping | 构建用户故事地图，按用户活动与任务组织 backlog。 | 发布前按用户流程切分 MVP 范围与版本边界。 |

| user-story-splitting | 将大故事/Epic 拆分为更小且可独立交付的故事以提升迭代流动性。 | 把跨月需求拆成每个 Sprint 可上线的增量。 |
| vp-cpo-readiness-advisor | 面向 Director 到 VP/CPO 的晋升转型辅导，覆盖准备、面试、入职与校准。 | 准备 VP 面试时演练组织设计与战略问答。 |
